# QSign Template Designer

## Purpose

Template Designer is a future offline desktop application for non-technical
administrators. It creates and validates QSign templates without direct file or
database editing.

It is an authoring tool, not part of the physician's document-signing flow.

## User goals

An administrator can:

- open representative PDFs;
- assign a document type and administrative metadata;
- select phrases and create matcher rules;
- test recognition scores against positive and negative samples;
- select textual anchors directly on a page;
- draw a desired signature rectangle;
- convert that drawing into an anchor-relative placement rule;
- validate and publish a versioned template;
- import, export, disable, or retire templates.

## Architectural position

Template Designer is a separate application composition using shared,
provider-neutral capabilities:

- PDF rendering for visual preview;
- document text/layout extraction for selectable text geometry;
- Template Engine for validation and lifecycle;
- Recognition Engine in simulation mode;
- Anchor and Placement Engines for preview;
- `TemplateRepository` for drafts and publication.

It does not write database tables directly from UI controls. It calls the
Template Engine, which applies validation and repository transactions.

The production QSign viewer does not gain Designer responsibilities.

## Main workflow

### 1. Create a draft

The user supplies template name, document type, description, and optional
validity scope. The Designer assigns a stable template identity and draft
version.

### 2. Add representative documents

The user opens one or more text-based PDF samples. The Designer shows whether a
usable text layer exists. Image-only documents are rejected because OCR is not
part of QSign.

Samples are test inputs and are not automatically stored in the template
repository. Storage requires an explicit privacy policy.

### 3. Define recognition rules

The page preview supports selecting text and creating:

- required literal phrases;
- exclusion phrases;
- weighted phrases;
- all/any/N-of-M term groups;
- approved regex rules through a guided form;
- page-count or metadata evidence.

The Designer displays each rule's scope, weight, and effect on the total score.

### 4. Simulate matching

For every sample, the user sees:

- gate outcomes;
- total template score;
- competing-template scores;
- threshold and ambiguity margin;
- exact rule evidence;
- `MATCHED`, `UNKNOWN`, or `AMBIGUOUS`.

Publishing is blocked when required positive samples fail or configured
negative samples match.

### 5. Define an anchor

The user clicks or selects visible text such as “L'interessato”. The Designer
stores the selected text expression, page scope, occurrence policy, and
surrounding context rules.

It does not store the clicked sample coordinates as the anchor definition.
Coordinates are retained only as sample validation evidence.

### 6. Define signature placement

The user draws a rectangle near the selected anchor. The Designer converts it
to:

- placement side and alignment;
- X and Y offsets relative to anchor geometry;
- width and height in PDF points;
- minimum margins.

The user may define ordered anchor-relative fallbacks. The preview is always
generated by the real Placement Engine, not independent UI calculations.

### 7. Validate

Validation combines:

- template schema validation;
- matcher and regex validation;
- positive and negative recognition fixtures;
- anchor uniqueness and context checks;
- page-bound and margin checks;
- workflow-reference checks;
- compatibility and checksum generation.

Errors block publication. Warnings require explicit acknowledgement and are
recorded in the audit trail.

### 8. Publish

Publication creates an immutable version transactionally. Activation is a
separate explicit action. A previous published version remains available for
historical analysis evidence and rollback.

## Proposed workspace

| Area | Purpose |
|---|---|
| Template navigator | Drafts, published versions, activation, and audit state |
| Sample documents | Positive, negative, and ambiguity test sets |
| PDF canvas | Rendered page with selectable text and placement overlay |
| Rule editor | Guided matcher, anchor, and placement forms |
| Evidence panel | Scores, gates, occurrences, and geometry diagnostics |
| Validation panel | Errors, warnings, compatibility, and publish readiness |

Advanced fields should be progressively disclosed. The default workflow should
favor literal text, grouped keywords, unique anchors, and simple below/above
placements.

## Safety and usability rules

- Never allow publication without validation.
- Never hide why a sample matched or failed.
- Never use drag position as an absolute document coordinate.
- Warn when selected anchor text is too common.
- Warn when two active templates are within the ambiguity margin.
- Require explicit confirmation before activation, retirement, import overwrite,
  or rollback.
- Autosave drafts, but never auto-publish.
- Preserve keyboard accessibility and usable zoom.
- Redact clinical content from general logs.

## Import and export

The Designer imports and exports a canonical JSON package containing:

- template definition and schema version;
- template version and engine compatibility;
- checksum;
- optional non-clinical test manifest;
- audit-safe descriptive metadata.

Import validates before creating a draft. Imported packages never become active
automatically.

## Local database interaction

The future default repository is planned as SQLite, but the Designer remains
repository-neutral. It uses Template Engine operations for drafts, publication,
activation, and audit.

Users never edit SQLite manually. Backup, migration, corruption recovery,
access control, and optional encryption policy must be designed before database
implementation.

## Out of scope

- AI-assisted template generation;
- OCR;
- automatic publishing;
- digital signatures;
- Wacom capture;
- certificate management;
- document upload or download;
- production workflow execution;
- direct SQL editor;
- client-specific code generation.
